# video-to-steps v1 (vts-v1) Design

## Summary

The video-to-steps app is a single-process FastAPI + HTMX application
that accepts a YouTube instructional-video URL and produces an ordered,
illustrated step-by-step guide. The pipeline chains yt-dlp download, VTT
caption parsing with rolling-repeat deduplication, pHash-deduplicated
frame extraction via ffmpeg, and two LLM passes (outline then refine),
tied together by a multimodal embedding match that selects which frames
to caption.

The central architectural idea — called Approach C — is to embed cheaply
across all frames using Jina's shared multimodal space (image and text in
the same vector space), use cosine similarity to select the top-k
"winner" frames per step, and only then invoke the expensive vision-LLM
to caption those ~30 winners. Those captions feed back into the LLM
refine pass to ground the polished step text in what is visibly on
screen. All provider concerns (embedder, vision captioner, text LLM) sit
behind protocol abstractions so that switching between three deployment
modes — Mode C (full cloud, v1 acceptance target), Mode A (local MLX
CLIP on Apple Silicon), and Mode B (hybrid) — requires only
environment-variable changes, with no code branching in the pipeline
itself.

## Definition of Done

**Primary deliverable.** A FastAPI + HTMX web app, runnable locally or on a
small Linux cloud host with `ffmpeg` installed, that accepts a YouTube
instructional-video URL and produces a result page of N ordered written
step-instructions, each illustrated with one or more frames extracted from
the video and visibly relevant to the step text.

**Success criteria.**

1. End-to-end smoke test passes in **Mode C** (cloud LLM + vision-LLM frame
   captioning + cloud text embeddings) on at least one short real
   instructional video, producing ≥3 steps each with ≥1 visibly-relevant
   illustrating frame.
2. Provider abstractions (LLM client, embedder factory with `Embedder`
   protocol) are in place so that switching cloud↔local↔hybrid is
   configuration only — no code changes.
3. Per-job manifest on disk records status, progress, error, and a per-job
   cost estimate (token counts × per-million rate for chat + vision +
   embeddings).
4. Server defaults to `127.0.0.1`; switching to `0.0.0.0` for cloud deploy
   is a single env var; README documents the reverse-proxy/VPN expectation.
5. README + `.env.example` document the Mode A (MLX CLIP) and Mode B
   (hybrid) paths well enough that a fresh Macbook session could try them,
   but these modes are not in the v1 smoke test.

**Out of scope for v1.**

- Tested Mode A (MLX CLIP) / Mode B (hybrid) end-to-end runs (documented
  and code-pathed, but unverified in v1)
- In-app authentication, multi-user, rate limiting, public hosting
- Multi-job queue, persistent task recovery across restarts, S3 storage
- Whisper fallback for captionless videos (fail with a clear error)
- Eval harness, A/B comparison, per-step feedback collection
- PDF / Markdown export, diff/refine UI
- Non-YouTube sources

## Acceptance Criteria

### vts-v1.AC1: End-to-end Mode C pipeline produces a usable result
- **vts-v1.AC1.1 Success:** A YouTube URL of a short instructional video produces a `steps.json` with ≥3 ordered steps, each with ≥1 frame.
- **vts-v1.AC1.2 Success:** Each step's `instruction` field is 1–3 second-person imperative sentences.
- **vts-v1.AC1.3 Success:** Each step's frames are visibly relevant to the instruction text (manual judgment on a checked-in test video).

### vts-v1.AC2: YouTube download and caption parsing
- **vts-v1.AC2.1 Success:** yt-dlp downloads MP4 at ≤720p and the English VTT auto-caption track for a valid YouTube URL.
- **vts-v1.AC2.2 Success:** VTT parsing returns a `list[Cue]` in temporal order.
- **vts-v1.AC2.3 Success:** `dedupe_rolling` collapses YouTube auto-caption rolling-repeat patterns into single Cues.
- **vts-v1.AC2.4 Failure:** When yt-dlp returns no captions and `WHISPER_FALLBACK` is disabled, the orchestrator writes `status="error"` with a clear human-readable message in `manifest.error`.

### vts-v1.AC3: Frame extraction with deduplication
- **vts-v1.AC3.1 Success:** `FixedFpsExtractor(fps=1.0, dedup=False)` extracts one frame per source second at ≤720p height.
- **vts-v1.AC3.2 Success:** `FixedFpsExtractor(fps=1.0, dedup=True, hamming_max=6)` produces strictly fewer frames than `dedup=False` on a cut-heavy test video; equal-or-fewer on any input.
- **vts-v1.AC3.3 Edge:** pHash dedup never drops the first frame.

### vts-v1.AC4: Provider abstractions are config-only switchable
- **vts-v1.AC4.1 Success:** `LLMClient` chats successfully against both qwen-studio `/chat` (single-text SSE) and an OpenAI-shape `/v1/chat/completions` endpoint, distinguishing them by auto-detect on response shape.
- **vts-v1.AC4.2 Success:** `LLMClient.chat()` always strips `<think>...</think>` blocks from returned text.
- **vts-v1.AC4.3 Success:** `JinaEmbedder.embed_images()` and `.embed_texts()` return L2-normalized float32 vectors of consistent dimensionality.
- **vts-v1.AC4.4 Success:** Switching `EMBED_BACKEND` from `jina_v4` to `mlx_clip` requires only an env-var change; `mlx_clip` raises a clear actionable error on a host without the dep installed.
- **vts-v1.AC4.5 Success:** `VisionCaptioner.caption()` returns a 1–2 sentence caption focused on actions, tools, and materials for a checked-in test still frame.

### vts-v1.AC5: LLM outline and refine passes
- **vts-v1.AC5.1 Success:** `llm_outline` returns ≥3 `StepOutline`s with non-overlapping time ranges that together cover the input transcript span.
- **vts-v1.AC5.2 Success:** `llm_outline` parses provider responses correctly when `response_format={"type":"json_object"}` is supported.
- **vts-v1.AC5.3 Failure:** `llm_outline` slice-fallback parses a fixture response containing prose around the JSON `[…]` block (unit-tested).
- **vts-v1.AC5.4 Success:** `llm_refine` produces 1–3 second-person imperative sentences per step, incorporating the winning-frame captions into the prompt.

### vts-v1.AC6: Frame-to-step matching
- **vts-v1.AC6.1 Success:** `match` restricts candidate frames per step to `[step.start - pad, step.end + pad]` before scoring.
- **vts-v1.AC6.2 Success:** `match` picks the top-k frames by cosine via `frame_emb @ step_emb` within the candidate window.
- **vts-v1.AC6.3 Edge:** When the candidate window is empty, `match` falls back to the single nearest frame to step midpoint.

### vts-v1.AC7: Cost tracking and manifest
- **vts-v1.AC7.1 Success:** At the end of a successful job, `meta.json.cost.total_usd` is non-zero and equals the sum of per-call costs from `pricing.py`.
- **vts-v1.AC7.2 Success:** `meta.json` is atomically written; the status-poll endpoint always returns parseable JSON, never a torn write.
- **vts-v1.AC7.3 Edge:** A configured model not present in `pricing.py` records zero for that line item and logs a startup warning; the job still completes.

### vts-v1.AC8: Server and result page
- **vts-v1.AC8.1 Success:** `GET /` renders the URL input form.
- **vts-v1.AC8.2 Success:** `POST /process` with a valid YouTube URL returns a 303 redirect to `/job/{id}` and spawns a pipeline task.
- **vts-v1.AC8.3 Failure:** `POST /process` with a non-YouTube URL returns 400 with a human-readable message.
- **vts-v1.AC8.4 Success:** `/job/{id}/status` returns an HTMX fragment showing current `status`, `progress`, and running `cost.total_usd`.
- **vts-v1.AC8.5 Success:** `/job/{id}/result` renders ordered steps each with caption-alt-texted thumbnails; small print shows `mode`, `cost.total_usd`, embedder name, vision model, and LLM model.

### vts-v1.AC9: Deployment story
- **vts-v1.AC9.1 Success:** Default `APP_HOST=127.0.0.1`; setting `APP_HOST=0.0.0.0` is the only change required for cloud binding.
- **vts-v1.AC9.2 Success:** README documents reverse-proxy/VPN expectation for `0.0.0.0` deployments and warns against open-internet exposure without one.
- **vts-v1.AC9.3 Success:** README documents Mode A (MLX CLIP on Macbook) and Mode B (hybrid) with full env-var blocks; explicitly notes both are untested in v1.

## Glossary

- **yt-dlp**: Command-line tool used to download YouTube video (MP4) and auto-caption tracks (VTT) from a URL.
- **VTT**: WebVTT subtitle file format; the auto-caption track downloaded from YouTube and parsed into timed text cues.
- **Cue**: A single timed caption segment parsed from the VTT file, carrying start time, end time, and text.
- **dedupe_rolling**: A post-parse step that collapses YouTube's rolling-repeat auto-caption pattern (where the same text appears across multiple overlapping cues) into single, clean Cues.
- **pHash**: Perceptual hash — a compact fingerprint of an image's visual content used to detect near-duplicate frames by comparing Hamming distance between hashes.
- **ffmpeg**: Command-line multimedia tool used to extract frames from the downloaded video at a fixed frame rate and scale them to ≤720p.
- **Jina embeddings**: A cloud embedding API (`JinaEmbedder`) that encodes both images and text into the same L2-normalized vector space, enabling direct cosine-similarity comparison between frame images and step-text strings.
- **CLIP / MlxClipEmbedder**: A multimodal model (run locally via the `mlx_clip` library on Apple Silicon) that similarly maps images and text into a shared embedding space; used in Mode A as the local alternative to Jina.
- **Approach C**: The pipeline design selected in brainstorming: embed all frames cheaply with a multimodal embedder, match to step text, then caption only the top-k winner frames with the vision LLM — reversing the PLAN.md approach of captioning every frame first.
- **Mode A / B / C**: Three deployment configurations differing in embedder and LLM provider. Mode C (cloud-only, Jina + cloud vision + cloud text LLM) is the v1 acceptance target; Modes A (local MLX CLIP, local LLM via qwen-studio) and B (MLX CLIP + cloud LLM) are documented but untested in v1.
- **qwen-studio**: A local LLM server (from `santhony/qwen-studio`) running at `127.0.0.1:8766`; the default LLM backend for Mode A, exposing a `/chat` endpoint with a dual-stream SSE response shape distinct from the OpenAI shape.
- **SSE (Server-Sent Events)**: The streaming response format used by both qwen-studio and OpenAI-compatible endpoints; `LLMClient` auto-detects which shape is in use from the response structure.
- **HTMX**: A JavaScript library that enables the job-status page to poll `/job/{id}/status` every two seconds and swap in updated progress fragments without a full page reload.
- **StepOutline**: An intermediate data type produced by the LLM outline pass, carrying index, start time, end time, and a brief description before the refine pass produces polished step text.
- **Manifest / meta.json**: The per-job record written atomically to disk, tracking pipeline status, progress, error, configuration snapshot, and running cost; the only file the server reads to serve status updates.

## Architecture

The system is a single-process FastAPI application that accepts a YouTube
URL, runs an in-process `asyncio` pipeline, and renders a result page of
ordered written instructions, each illustrated by frames from the source
video.

Three deployment modes share the same code; differences live entirely in
configuration:

| Mode | Embedder | Vision captioner | Text LLM | v1 status |
|---|---|---|---|---|
| C — Cloud (default) | `JinaEmbedder` | cloud (gpt-4o-mini or similar) | cloud (DeepSeek v4-flash or similar) | Acceptance target |
| A — Local (Macbook) | `MlxClipEmbedder` | local or cloud | local (qwen-studio @ 127.0.0.1:8766) | Documented, untested |
| B — Hybrid | `MlxClipEmbedder` | cloud | cloud | Documented, no separate code path |

### Approach: cheap embed match, expensive caption of winners

The cloud-mode pipeline (selected in brainstorming as Approach C):

```
YouTube URL
  ├─► yt-dlp                          ─► video.mp4 + captions.vtt
  ├─► parse_vtt + dedupe_rolling      ─► list[Cue]
  ├─► FrameExtractor (FixedFps+dedup) ─► frames/####.jpg + list[Frame]
  ├─► Embedder.embed_images(all)      ─► frame_embeddings.npy
  ├─► LLM Pass 1 (outline)            ─► list[StepOutline]
  ├─► Embedder.embed_texts(briefs)    ─► step_embeddings
  ├─► match(steps, frames, embs)      ─► per-step top-k Frame winners
  ├─► VisionCaptioner.caption(winners) ─► ~30 captions cached
  ├─► LLM Pass 2 (refine) — brief + cue snippet + winner captions
  │                                   ─► list[Step] with polished text
  └─► write steps.json; manifest.status="done"
```

Cheap matching across all frames in Jina's shared multimodal space carries
the search burden. The expensive vision-LLM caption call is reserved for
the ~30 frames that actually end up on the result page, where it grounds
the LLM refine pass with what is visibly happening on screen.

### Provider abstractions

Four protocols decouple the pipeline from concrete providers; pipeline code
never branches on backend.

```python
class LLMClient:
    async def chat(self, messages, *, max_tokens=None,
                   response_format=None) -> ChatResult
    # ChatResult = (text, prompt_tokens, completion_tokens)

class VisionCaptioner:
    async def caption(self, image: Path) -> CaptionResult
    # CaptionResult = (text, prompt_tokens, completion_tokens)

class Embedder(Protocol):
    def name(self) -> str
    async def embed_images(self, paths: list[Path]) -> EmbedResult
    async def embed_texts(self, texts: list[str]) -> EmbedResult
    # EmbedResult = (vectors: np.ndarray float32 (n,d) L2-normalized,
    #                billable_tokens: int)

class FrameExtractor(Protocol):
    def name(self) -> str
    def extract(self, video: Path, out_dir: Path) -> list[Frame]
```

Factory functions in each provider module read `config.Settings` and return
the concrete instance. Every chat, caption, and embed call returns
billable-token counts so the orchestrator can sum them into the per-job
`cost.total_usd` without re-parsing responses.

### Concurrency budgets

Configured, not hard-coded:

- `JINA_EMBED_BATCH = 64` — images per Jina request.
- `REFINE_MAX_IN_FLIGHT = 4` — step-refine fanout.
- `CAPTION_MAX_IN_FLIGHT = 16` — vision-caption fanout (only over winners,
  so ~30 calls per video — well under any provider's rate limit).

### Data-flow boundaries

All artifacts for a job live in `data/jobs/{job_id}/`: `meta.json`
(`Manifest`), `video.mp4`, `captions.vtt`, `frames/####.jpg`,
`frame_embeddings.npy`, `outline.json`, `frame_captions.json` (winners
only), `steps.json`. `pipeline/storage.py` provides atomic write helpers
(temp + rename) so the status-poll endpoint never observes a torn write.

The server (`server.py`) reads only from `meta.json` and `steps.json` and
serves frame thumbnails directly from disk. No in-memory shared state
between requests.

## Existing Patterns

This is a greenfield repository (`santhony/video-to-steps`); the only prior
content is `PLAN.md` and `README.md`, both authored by the same person as
this design. There is no existing application code to follow.

The design follows the structure of `PLAN.md` with deliberate divergences,
each justified by brainstorming research:

| `PLAN.md` (May 2026 draft) | This design (`vts-v1`) | Reason |
|---|---|---|
| `EMBED_BACKEND=vision_caption` for Mode C — caption every frame, then text-embed | `JinaEmbedder` for matching, vision-caption only the top-k winners per step | Jina's multimodal `/v1/embeddings` shape didn't exist when `PLAN.md` was authored. Using it cuts ~1770 vision-LLM calls per 30-min video and makes the refine pass visually grounded by reserving captioning for the frames that matter. |
| Fixed 1 fps `extract_frames` function | `FrameExtractor` protocol; `FixedFpsExtractor(dedup=True)` default; `SceneChangeExtractor` stub | Pluggable strategy enables iteration across video styles (cooking, screencasts, knot-tying) without code changes. |
| Single `LLMClient` handles chat with optional `image_url` parts | Separate `LLMClient` + `VisionCaptioner` | The caption-winners step has distinct config: tighter `max_tokens`, narrower prompt, and may be a different provider from the text LLM. |
| `Manifest` records `status`, `progress`, `error` | `Manifest` adds `cost: CostBreakdown` and `config_snapshot` | Iteration across configs and providers needs cost visibility and per-run provenance. |
| All three modes are v1 acceptance targets (§16) | Mode C only is v1 acceptance; A/B documented and code-pathed but untested | The developer's primary machine has no Apple Silicon; cloud-first dev is honest about what actually gets exercised. |

External references that informed the design (not in-repo):

- `santhony/qwen-studio` (GitHub) — defines the local-server defaults:
  `LLM_BASE_URL=http://127.0.0.1:8766`, `LLM_PATH_CHAT=/chat`, dual-stream
  response shape.
- `specstoryai/keyframe-blogger` (GitHub, 2024–2025) — closest published
  precedent for a transcript-plus-keyframe-plus-LLM video-to-text pipeline.
  Used as a sanity-check reference, not a code dependency.
- DeepSeek v4 `<think>` block leakage in `reasoning_content` (project
  memory): requires `LLM_MAX_TOKENS ≥ 1024` and `<think>...</think>`
  stripping in the chat client.

consume. Each phase ends with a verifiable acceptance state. Acceptance
Criteria identifiers (§Acceptance Criteria) are listed per phase.

### Phase 2: Provider implementations

**Goal:** Three concrete provider classes that talk to real APIs and
report token usage.

**Components:**

- `providers/llm.py::LLMClient` — POSTs to `base_url + path`; dual-stream

- `providers/embed_jina.py::JinaEmbedder` — `/v1/embeddings` for both

- `providers/embed_mlx_clip.py::MlxClipEmbedder` — import-guarded

- Smoke scripts: `scripts/smoke_llm.py`, `scripts/smoke_vision.py`,

**ACs covered:** `vts-v1.AC4.1`, `vts-v1.AC4.2`, `vts-v1.AC4.3`,

### Phase 3: Source-to-frames pipeline

**Goal:** Given a URL, produce a deduplicated frame set and a parsed
caption track.

**Components:**

- `pipeline/download.py::download_video_and_captions(url, job_dir)
  -> tuple[Path, Path | None]` — yt-dlp wrapper with
  `writeautomaticsub=True, subtitleslangs=["en"], subtitlesformat="vtt"`,
  `format="best[ext=mp4][height<=720]/best[height<=720]/best"`.
- `pipeline/captions.py::parse_vtt`,
  `pipeline/captions.py::dedupe_rolling` — webvtt-py based; dedupe
  collapses YouTube auto-caption rolling repeats.
- `pipeline/frames.py::FixedFpsExtractor(fps=1.0, dedup=True,
  hamming_max=6)` — ffmpeg call with
  `-vf "fps={fps},scale=-2:720" -qscale:v 4`, then an in-Python pHash
  pass that drops frames within Hamming-distance-N of the previous kept
  frame.
- `pipeline/frames.py::SceneChangeExtractor` — interface-only stub;
  raises `NotImplementedError` with a v2 roadmap note.
- Unit tests: VTT parsing on a checked-in sample fragment; pHash dedup on
  a synthetic 4-image set (two near-duplicates, two unique).

**Dependencies:** Phase 1 (types, storage).

**Done when:** Unit tests pass. A short ad-hoc script can download one
~3-minute YouTube video, parse its VTT into `list[Cue]`, and extract a
deduplicated frame set to a temp directory.

**ACs covered:** `vts-v1.AC2.1`, `vts-v1.AC2.2`, `vts-v1.AC2.3`,
`vts-v1.AC3.1`, `vts-v1.AC3.2`, `vts-v1.AC3.3`.

<!-- START_PHASE_4 -->
### Phase 4: LLM passes

**Goal:** Outline and refine LLM passes with structured-JSON robustness.

**Components:**

- `prompts/outline.md` — system + user template that takes the deduped
  transcript (one `[MM:SS] text` line per cue) and asks for a
  `[{index, start, end, brief}, ...]` JSON array.
- `prompts/refine.md` — system + user template that takes a single step's
  brief, the cue snippets covering its time range, and (in Phase 5) the
  captions of its winning frames, and produces 1–3 second-person
  imperative sentences.
- `prompts/vision_caption.md` — vision-LLM system prompt: "Describe what
  is shown in this still frame in one detailed sentence focused on
  actions, tools, and materials. Do not speculate beyond the image."
- `pipeline/llm_outline.py::llm_outline(cues, llm)
  -> tuple[list[StepOutline], ChatUsage]` — single LLM call with
  `response_format={"type":"json_object"}` when supported; manual
  `[…]` slice fallback ("locate first `[`, last matching `]`, parse")
  when not.
- `pipeline/llm_refine.py::llm_refine(steps, cues, winners_by_step,
  captions, llm, max_in_flight=4) -> tuple[list[Step], ChatUsage]` —
  per-step concurrent fanout with
  `asyncio.Semaphore(REFINE_MAX_IN_FLIGHT)`.
- Tests: given a hand-written cue list with known step boundaries,
  outline produces ≥3 sensible steps; given a step + cue snippet +
  (in this phase, empty) caption list, refine produces well-formed
  sentences; slice-fallback exercised by feeding a fixture response with
  prose surrounding the JSON.

**Dependencies:** Phase 2 (`LLMClient`), Phase 3 (`Cue` type).

**Done when:** Tests pass against the configured cloud LLM endpoint.

**ACs covered:** `vts-v1.AC5.1`, `vts-v1.AC5.2`, `vts-v1.AC5.3`,
`vts-v1.AC5.4`.
<!-- END_PHASE_4 -->

<!-- START_PHASE_5 -->
### Phase 5: Match, caption-winners, orchestrator

**Goal:** End-to-end pipeline run produces `steps.json` and updates
`meta.json` with running and final cost.

**Components:**

- `pipeline/match.py::match(steps, frames, frame_emb, step_emb, pad_sec,
  top_k) -> list[list[Frame]]` — pure function; per step, restrict to
  frames in `[start - pad, end + pad]`; cosine score via
  `frame_emb @ step_emb`; pick top-k; nearest-to-midpoint fallback when
  candidate window is empty. Easy to unit-test with hand-crafted vectors.
- `pipeline/caption_winners.py::caption_winners(winners_by_step, job_dir,
  captioner, max_in_flight=16) -> tuple[dict[int, str], CaptionUsage]` —
  vision-LLM fanout across all unique winning frames; caches to
  `frame_captions.json`; tolerates per-frame failures (frame's caption
  becomes `None`).
- `pipeline/pipeline.py::run_job(job_id, url, settings, jobs_root)` —
  orchestrator chaining all stages; wraps each in
  `manifest.update(status=..., progress=...)`; accumulates token counts,
  passes them through `pricing.compute_cost()`, writes
  `manifest.cost`; outer `try/except` writes `error=str(exc)` and
  re-raises into the task.
- Tests: `match` unit tests (window restriction, top-k correctness,
  empty-window fallback); end-to-end integration test on a short
  checked-in YouTube URL that asserts `meta.json.status == "done"`,
  `len(steps) >= 3`, every step has `len(frames) >= 1`, and
  `meta.json.cost.total_usd > 0`.

**Dependencies:** Phases 2, 3, 4.

**Done when:** Integration test passes on one short real instructional
video. Manifest shows non-zero, plausible cost.

**ACs covered:** `vts-v1.AC1.1`, `vts-v1.AC1.2`, `vts-v1.AC1.3`,
`vts-v1.AC2.4`, `vts-v1.AC6.1`, `vts-v1.AC6.2`, `vts-v1.AC6.3`,
`vts-v1.AC7.1`, `vts-v1.AC7.2`, `vts-v1.AC7.3`.

## Additional Considerations

**Error handling philosophy.** Surface, don't paper over. The
orchestrator's outer `try/except` writes `error=str(exc)` to the
manifest and re-raises into the task. The result page reads
`meta.json.error` and shows it verbatim. Three specific cases get
cleaner UX (clear status-page messages): no YouTube captions returned by
yt-dlp ("This video has no captions. Whisper fallback is on the v2
roadmap."), per-frame vision-model rejection (skip frame, caption=None,
refine pass tolerates), and `<think>` block leakage from reasoning-tier
models (`LLMClient` strips unconditionally; outline pass pairs
`response_format=json_object` with a slice fallback).

**Cost reporting.** Every chat, caption, and embed call returns
billable-token counts. The orchestrator sums them after each stage and
writes the running total into `meta.json.cost`. The status page polls
and displays the running cost; the result page shows the final cost in
small print at the top of the result, alongside `config_snapshot`
essentials. Iterating on prompts and configs without cost visibility is
what makes this design viable for "robust across video styles" while
keeping costs at personal-use scale.

**Approach-C quality escape hatch.** Jina image-embedding quality on
action-verb-heavy step text is unproven for instructional video. If
matching feels noisy in practice, the fix is to write a second
`Embedder` implementation (`providers/embed_caption_then_text.py`) that
runs the vision LLM on all frames and then text-embeds the captions —
the original `PLAN.md` Approach A. Pipeline code does not change; only
the `EMBED_BACKEND` env var. This is documented explicitly in the
README.

**Mode A and B in v1.** Both modes' code paths exist (the MLX CLIP
embedder behind an import guard; local-LLM defaults in `.env.example`)
but they are not in the v1 smoke-test bar. The README documents how to
try them. If `mlx_clip` install fails on the Macbook, the user can fall
back to Mode C against a cloud provider with no code changes.

**Future extensibility.** The `FrameExtractor` protocol enables
`SceneChangeExtractor` for cut-heavy content as a v2 implementation
behind a config flag. The `Embedder` protocol enables the
caption-then-embed fallback above. The `LLMClient`/`VisionCaptioner`
split enables routing the cheap step-by-step LLM work to one provider
(e.g., DeepSeek) and the more expensive vision captioning to a
different one (e.g., OpenAI or Anthropic) without touching pipeline
code.
